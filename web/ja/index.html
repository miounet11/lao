AI エージェント、エンジニア、MCP クライアント向けの認証済みブラウザ自動化

ログイン済みの Chrome をそのまま API にする。

iatlas-browser は新しいヘッドレスブラウザから始めません。いま使っている Chrome セッションをそのまま使うので、ログイン状態、Cookie、タブ、ページ状態を保ったまま 自動化できます。miaoda.vip は公開かつリモート安全な処理だけを担う、 より小さなホスト API サブセットです。

Cookie の書き出し不要 CLI + MCP + ローカル HTTP を 1 つの実行系で共有 ホスト API は安全なサブセットだけ
install.sh を取得 アダプター一覧 Open API docs 拡張機能 zip GitHub

3 分で最初の成功

curl -fsSL https://miaoda.vip/install.sh | bash
iatlas-browser daemon
iatlas-browser doctor
iatlas-browser snapshot -i
  • ~/.iatlas-browser/src にインストールしてローカル実行系を構築
  • CLI、daemon、MCP、拡張機能、設定例をまとめて用意
  • 認証状態を作り直さず、最初の実ブラウザ snapshot まで進める
実セッション 実際に使っているブラウザの Cookie、タブ、ページ文脈をそのまま扱います。
統一ランタイム CLI、MCP、ローカル HTTP が同じ実行モデルを共有します。
境界が明確 ブラウザ依存の処理はローカル、公開安全な処理だけをホスト側に送ります。

違い

多くの自動化はクリーンなブラウザから始まる。iatlas-browser はあなたのブラウザから始まる。

この違いが製品の中心です。認証済みサイト、社内ダッシュボード、SNS、実ブラウザ前提のエージェント処理で 最初の成功までの距離が大きく変わります。

作業 通常のヘッドレス iatlas-browser
すでにログイン済みのサイトを使う 別環境で認証や状態を再構築する必要がある いま使っている Chrome のセッションをそのまま使う
実ページ上で壊れたフローを調べる 実ブラウザと自動化ブラウザを行き来する 同じブラウザ文脈の中で確認と操作を行える
MCP エージェントにブラウザを渡す 別のサンドボックスを露出しがち CLI、MCP、HTTP を共有するローカル実行系を渡せる
公開 URL の遠隔取得を行う 不要にブラウザ自動化へ寄りやすい miaoda.vip の小さなホスト API サブセットへ分離できる

Open API

メールで API キーを発行

https://miaoda.vip/v1/ でホスト API を利用できます。メールにひもづく API キーごとに、 1 日あたり 1000 リクエスト まで使えます。

API 概要

  • 認証: `Authorization: Bearer <apiKey>`
  • 上限: 1 日 1000 リクエスト
  • エンドポイント: `POST /v1/register`, `GET /v1/usage`, `POST /v1/open`

何が便利か

ログイン済みサイトの自動化に強い

既存の認証状態を再利用

社内ツール、管理画面、SaaS ダッシュボードなど、再ログインが面倒なサイトに向いています。

AI ツールへの接続が簡単

MCP モードとローカル API があるため、エージェントから実ブラウザを扱う入口を作りやすいです。

最初の手順

セットアップの流れ

1. インストール

curl -fsSL https://miaoda.vip/install.sh | bash

2. 拡張機能を読み込む

chrome://extensions/

~/.iatlas-browser/extension を指定します。

3. デーモン起動

iatlas-browser daemon
iatlas-browser doctor

4. コマンド実行

iatlas-browser open https://example.com
iatlas-browser snapshot -i

接続方法

CLI、MCP、ローカル API

CLI

ブラウザ操作、スナップショット、クリック、ネットワーク確認などを直接実行できます。

MCP

Cursor や Claude Desktop 用に使える MCP 設定を出力できます。

cursor.json

API

ローカル HTTP API を使って外部スクリプトから制御できます。

examples.sh
