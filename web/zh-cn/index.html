面向 AI Agent、工程师、MCP 客户端的登录态浏览器自动化

把你登录中的 Chrome 直接变成 API。

iatlas-browser 不从一个全新的无头浏览器开始，而是直接接管你已经在使用的 Chrome 会话。真实标签页、真实 Cookie、真实页面状态、真实登录上下文都能直接复用。miaoda.vip 只承载较小的托管 API 子集，真正依赖浏览器上下文的工作仍然应该走本地运行时。

不需要导出 Cookie CLI + MCP + 本地 HTTP 共用一套运行时 Direct CDP 的 ref 现在可以跨命令复用 托管 API 只做远程安全的子集

本轮本地升级

没有扩展时，direct CDP 也能跑通常见的 snapshot 驱动流程。

只要开启 remote debugging，daemon 就会保留 direct-CDP 快照里的 refs。后续命令可以继续按 ref 点击、悬停、填充、追加输入和读取文本，extensionless 路径不再只是只读检查。

Direct ref 持久化

先执行一次 snapshot，后续 direct CDP 下继续复用这些 ref 做 click、hover、fill、type、get text。

浏览器识别更全

iatlas-browser doctor 现在会识别 Chrome、Beta、Dev、Canary、Chromium、Brave、Arc，并输出 remote debugging 启动提示。

MCP 连续性更强

即使暂时走 direct CDP 兜底路径，常见的 MCP 浏览器流程也不会立刻中断。

Open API

填写邮箱，立即领取 API Key

现在可以直接通过 https://miaoda.vip/v1/ 调用托管接口。每个邮箱绑定一个 API Key， 默认每天提供 1000 次 open 请求。

接口说明

  • 认证：`Authorization: Bearer <apiKey>`
  • 配额：每邮箱每天 1000 次
  • 端点：`POST /v1/register`、`GET /v1/usage`、`POST /v1/open`
  • 目标：公开的 `http` / `https` 地址

为什么这个工具有价值

核心逻辑：不是模拟浏览器，而是接管你真实在用的浏览器

直接复用登录态

很多网站的难点不是 DOM 操作，而是登录态、风控、前端动态校验和页面上下文。iatlas-browser 的重点就是跳过“重新登录”和“重建环境”这一步。

给终端和 Agent 提供统一接口

你可以用命令行直接操作，也可以走 MCP 接给 Cursor / Claude Desktop，还可以通过本地 HTTP API 接给你自己的服务或脚本。

快速开始

从安装到第一次执行命令

1. 安装

curl -fsSL https://miaoda.vip/install.sh | bash

2. 加载扩展

chrome://extensions/

选择 ~/.iatlas-browser/extension 目录。

3. 启动本地运行时

iatlas-browser daemon
iatlas-browser doctor

4. 立即使用

iatlas-browser open https://example.com
iatlas-browser snapshot -i
iatlas-browser mcp-config cursor

三种接入方式

CLI、MCP、API 都能直接用

CLI 浏览器命令

适合命令行自动化、调试、快照、点击、抓取、网络查看等场景。

iatlas-browser snapshot -i
iatlas-browser click @3

MCP 浏览器服务

适合接到 Cursor、Claude Desktop 或支持 MCP 的 Agent 运行时。

iatlas-browser mcp-config cursor

差异点

和传统浏览器自动化工具相比，最大的区别是什么

对比项 常规无头浏览器 iatlas-browser
登录态 通常需要重新构建 直接使用现有会话
页面上下文 独立环境 真实用户环境
Agent 接入 需要自建桥接层 内置 MCP
服务调用 通常需另起服务 内置本地 HTTP API

常见问题

安装前通常会问什么

它适合登录复杂的网站吗？

适合，这正是它最有价值的地方之一。

可以给 Agent 用吗？

可以，项目内置了 MCP 方式，也有本地 API。

是不是云端浏览器？

不是。它是本地桥接层，连接你的本地工具和你的本地浏览器。
